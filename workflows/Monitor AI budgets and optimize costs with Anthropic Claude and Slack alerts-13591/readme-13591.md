# Monitor AI budgets and optimize costs with Anthropic Claude and Slack alerts

## 1. Workflow Overview

**Purpose:**  
This workflow monitors AI infrastructure spend, uses Anthropic Claude-based agents to analyze costs and propose optimizations, then escalates via **Slack alerts** for urgent situations or **executive email reports** otherwise. It also writes an audit trail to n8n **Data Tables**.

**Primary use cases:**
- Finance/FinOps teams tracking multi-model LLM usage and avoiding budget overruns
- Engineering/Operations teams receiving automated escalation and mitigation steps
- Building cost intelligence pipelines where recommendations are generated by specialist agents

### 1.1 Scheduled Execution & Configuration
Runs on a schedule (hourly) and loads workflow parameters (budget limits, thresholds, notification endpoints).

### 1.2 Metrics Ingestion (Mock Data)
Generates a 30-day dataset of usage/cost/latency by model and workload, including summaries and a budget utilization view.

### 1.3 Cost Intelligence (LLM Analysis + Persistence)
Claude analyzes metrics, returns a structured JSON cost analysis, and stores key fields in a Data Table.

### 1.4 Budget Status Routing → Optimization Coordination (Multi-agent)
Routes by `budgetStatus` into an Optimization Coordinator agent that calls 3 specialist tools:
- Budget alert generation
- Routing optimization recommendations
- Cost projection calculator (code tool)

### 1.5 Action-Type Routing → Notification + Persistence
Routes by `actionType` to Slack (alerts) or Email (reports), merges notification results, and stores actions in a second Data Table.

**Entry point:** `Schedule Trigger` (single entry).  
**Sub-workflows invoked:** none (all logic is within this workflow).  
**External integrations:** Anthropic (Claude), Slack, Email (SMTP/Gmail via Email Send node), n8n Data Tables.

## 2. Block-by-Block Analysis

### Block 1 — Scheduled Execution & Workflow Configuration

**Overview:**  
Triggers the workflow periodically and defines configurable parameters (budget thresholds and notification targets) used later by Slack and email nodes.

**Nodes involved:**
- Schedule Trigger
- Workflow Configuration

#### Node: Schedule Trigger
- **Type / role:** `scheduleTrigger` — starts workflow automatically.
- **Configuration:** Runs every **1 hour** (`interval` with field `hours`).
- **Connections:**
  - Output → `Workflow Configuration`
- **Edge cases / failures:**
  - Misconfigured timezone or schedule frequency (operational rather than execution error).
- **Version notes:** typeVersion `1.3`.

#### Node: Workflow Configuration
- **Type / role:** `set` — central configuration variables.
- **Key fields set:**
  - `monthlyBudgetLimit` = 10000
  - `criticalThreshold` = 0.9
  - `warningThreshold` = 0.75
  - `slackChannel` = placeholder Slack channel ID
  - `executiveEmail` = placeholder recipient email
- **Important behavior:** `includeOtherFields: true` (keeps incoming fields).
- **Connections:**
  - Input ← `Schedule Trigger`
  - Output → `Generate Mock Metrics Data`
- **Edge cases / failures:**
  - Placeholder values not replaced will cause downstream notification failures (Slack channel ID invalid, email missing).
- **Version notes:** typeVersion `3.4`.

### Block 2 — Metrics Ingestion (Mock Metrics Generator)

**Overview:**  
Produces realistic synthetic metrics for the last 30 days across multiple LLM models and workload types, including cost, tokens, latency, and error/cache rates.

**Nodes involved:**
- Generate Mock Metrics Data

#### Node: Generate Mock Metrics Data
- **Type / role:** `code` — generates mock dataset and summary JSON.
- **Key logic (interpreted):**
  - Models list with per-1k token input/output pricing (e.g., `gpt-4`, `claude-3-*`).
  - Workloads: `customer-support`, `content-generation`, `data-analysis`, `code-generation`, `summarization`.
  - Generates per day (30 days) × model × workload metrics:
    - requests, input/output/total tokens
    - inputCost/outputCost/totalCost
    - latency avg/p95/p99, errorRate, cacheHitRate
  - Builds aggregates:
    - `summary.totalCost`, `avgCostPerRequest`, `avgCostPerDay`
    - `costByModel`, `costByWorkload`
    - `budget` object: **hardcoded monthly = 50000** and utilization fields
- **Connections:**
  - Input ← `Workflow Configuration`
  - Output → `Cost Intelligence Agent`
- **Edge cases / failures:**
  - CPU/memory load: 30 days × 5 models × 5 workloads = 750 records; usually fine, but can grow if expanded.
  - The workflow configuration’s `monthlyBudgetLimit` is **not used** here; budget is hardcoded to 50000, potentially causing mismatch with configured limits.
- **Version notes:** typeVersion `2`.

### Block 3 — Cost Intelligence (Claude Analysis + Structured Parsing + Storage)

**Overview:**  
A Claude agent analyzes the metrics and outputs a structured cost intelligence report (budget status, anomalies, optimization opportunities, risk, projection). Key fields are persisted to a Data Table for history/audit.

**Nodes involved:**
- Cost Intelligence Agent
- Anthropic Model - Cost Intelligence
- Cost Analysis Output Parser
- Store Cost Analysis

#### Node: Cost Intelligence Agent
- **Type / role:** `@n8n/n8n-nodes-langchain.agent` — orchestrates the LLM reasoning step.
- **Prompt input:**  
  `Analyze the following AI infrastructure metrics: {{ JSON.stringify($json) }}`
- **System message:** Cost Intelligence Agent responsibilities (usage, latency bottlenecks, distribution, anomalies, budget utilization, optimization, risk).
- **Output parsing:** `hasOutputParser: true` → enforced schema via `Cost Analysis Output Parser`.
- **Connections:**
  - Input ← `Generate Mock Metrics Data`
  - AI model input ← `Anthropic Model - Cost Intelligence` (via `ai_languageModel`)
  - Output parser input ← `Cost Analysis Output Parser` (via `ai_outputParser`)
  - Main outputs:
    - Output 1 → `Route by Budget Status`
    - Output 2 → `Store Cost Analysis`
- **Edge cases / failures:**
  - Anthropic auth errors/quota limits.
  - Output parser failures if the model returns JSON that does not match the schema example shape/types.
  - Large JSON stringification could increase token usage; with real metrics, consider summarizing before sending.
- **Version notes:** typeVersion `3.1`.

#### Node: Anthropic Model - Cost Intelligence
- **Type / role:** `lmChatAnthropic` — Claude chat completion provider.
- **Model:** `claude-3-7-sonnet-20250219`
- **Temperature:** `0.2` (more deterministic).
- **Credentials:** “Anthropic account” (Anthropic API).
- **Connections:**
  - Output → `Cost Intelligence Agent` (as `ai_languageModel`)
- **Edge cases / failures:**
  - Incorrect API key, model not available in region/account, transient network timeouts.
- **Version notes:** typeVersion `1.3`.

#### Node: Cost Analysis Output Parser
- **Type / role:** `outputParserStructured` — validates/coerces LLM output into structured JSON.
- **Schema example fields:** `totalCost`, `budgetUtilization`, `budgetStatus`, breakdowns, anomalies, opportunities, `riskLevel`, `projectedMonthEndCost`, etc.
- **Connections:**
  - Output → `Cost Intelligence Agent` (as `ai_outputParser`)
- **Edge cases / failures:**
  - If agent returns `budgetStatus` values not matching downstream Switch expectations (critical/warning/normal), routing may go to fallback.
- **Version notes:** typeVersion `1.3`.

#### Node: Store Cost Analysis
- **Type / role:** `dataTable` — persists cost analysis snapshot.
- **Target table:** `CostAnalysisHistory`
- **Mapped columns:**
  - `riskLevel`, `timestamp`, `totalCost`, `budgetStatus`, `budgetUtilization`, `projectedMonthEndCost`
- **Connections:**
  - Input ← `Cost Intelligence Agent`
- **Edge cases / failures:**
  - Data Table not created / renamed (`CostAnalysisHistory` missing).
  - Type mismatch if some fields are null/undefined (depending on Data Table column definitions).
- **Version notes:** typeVersion `1.1`.

**Sticky note content (applies here):**
- “Cost Intelligence Agent — Claude classifies budget status across Critical, Warning, Review, and Feedback tiers…”
  - Note: the actual Switch later only routes for `critical`, `warning`, `normal`.

### Block 4 — Budget Status Routing → Optimization Coordination (Multi-agent Tools)

**Overview:**  
Routes the cost analysis by `budgetStatus` and sends it to a coordinator agent that calls three specialist tools (budget alert, routing recommendations, cost projection) and returns a structured plan including `actionType` (“alert” vs “report”).

**Nodes involved:**
- Route by Budget Status
- Optimization Coordinator Agent
- Anthropic Model - Coordinator
- Coordinator Output Parser
- Budget Alert Agent Tool
- Anthropic Model - Budget Alert
- Budget Alert Output Parser
- Routing Recommendation Agent Tool
- Anthropic Model - Routing
- Routing Output Parser
- Cost Projection Calculator Tool

#### Node: Route by Budget Status
- **Type / role:** `switch` — routes by `budgetStatus`.
- **Rules (string equals):**
  - “Critical” if `{{$json.budgetStatus}} == "critical"`
  - “Warning” if equals `"warning"`
  - “Normal” if equals `"normal"`
- **Fallback output:** `extra`
- **Connections:**
  - Input ← `Cost Intelligence Agent`
  - Output (all cases) → `Optimization Coordinator Agent` (only one connection shown; functionally, all statuses flow there in this JSON)
- **Edge cases / failures:**
  - If `budgetStatus` values differ (e.g., “over_budget”, “review”, “feedback”), it will go to fallback; but since fallback is still connected onward (in this workflow it routes to the same coordinator), the label becomes informational only unless additional paths are added.
- **Version notes:** typeVersion `3.4`.

#### Node: Optimization Coordinator Agent
- **Type / role:** `agent` — orchestrates calls to specialist tools and produces a coordinated plan.
- **Prompt input:** `Cost analysis data: {{ JSON.stringify($json) }}`
- **System message:** instructs calling:
  1) Budget Alert Agent Tool  
  2) Routing Recommendation Agent Tool  
  3) Cost Projection Calculator Tool  
  and synthesizing an action plan + notification strategy:
  - CRITICAL → Slack alerts
  - WARNING/NORMAL → executive reports
- **Output parsing:** `hasOutputParser: true` via `Coordinator Output Parser`.
- **Tool connections:**
  - Receives tool definitions from:
    - `Budget Alert Agent Tool` (ai_tool)
    - `Routing Recommendation Agent Tool` (ai_tool)
    - `Cost Projection Calculator Tool` (ai_tool)
  - Receives LLM from `Anthropic Model - Coordinator`
- **Connections:**
  - Input ← `Route by Budget Status`
  - Output → `Route by Action Type`
- **Edge cases / failures:**
  - If tools return unexpected shapes, coordinator may produce invalid JSON → output parser fail.
  - Tool invocation depends on correct `$fromAI(...)` usage inside tool nodes; misbinding can yield empty inputs.
- **Version notes:** typeVersion `3.1`.

#### Node: Anthropic Model - Coordinator
- **Type / role:** `lmChatAnthropic`
- **Model:** `claude-3-7-sonnet-20250219`
- **Temperature:** `0.3`
- **Credentials:** Anthropic
- **Connections:** Output → `Optimization Coordinator Agent` (`ai_languageModel`)
- **Failure types:** same as other Anthropic nodes.
- **Version notes:** typeVersion `1.3`.

#### Node: Coordinator Output Parser
- **Type / role:** `outputParserStructured`
- **Schema example:** expects fields like:
  - `actionType` (“alert” or “report”)
  - `priority`, `coordinatedActions[]`
  - summaries (`budgetAlertSummary`, `routingRecommendationSummary`, `costProjection`)
  - `notificationChannel`, `executiveSummary`
- **Connections:** Output → `Optimization Coordinator Agent` (`ai_outputParser`)
- **Edge cases:** if `actionType` not exactly “alert” or “report”, later Switch falls back.
- **Version notes:** typeVersion `1.3`.

#### Node: Budget Alert Agent Tool
- **Type / role:** `agentTool` — a callable specialist agent exposed as a tool to the coordinator.
- **Input text expression:**  
  `{{ $fromAI("costAnalysis", "Cost analysis data from Cost Intelligence Agent", "json") }}`
- **System message:** generate severity, actions, stakeholders, projected impact, emergency controls.
- **Output parsing:** yes, via `Budget Alert Output Parser`.
- **Model:** via `Anthropic Model - Budget Alert`.
- **Connections:**
  - Tool output → `Optimization Coordinator Agent` (`ai_tool`)
  - Receives `ai_languageModel` from `Anthropic Model - Budget Alert`
  - Receives `ai_outputParser` from `Budget Alert Output Parser`
- **Edge cases / failures:**
  - `$fromAI("costAnalysis", ...)` requires that the coordinator actually provides that named context; if not, tool receives empty/invalid JSON.
- **Version notes:** typeVersion `3`.

#### Node: Anthropic Model - Budget Alert
- **Type / role:** `lmChatAnthropic`
- **Model:** `claude-3-7-sonnet-20250219`, temperature `0.2`
- **Connections:** → `Budget Alert Agent Tool` (`ai_languageModel`)
- **Version notes:** typeVersion `1.3`.

#### Node: Budget Alert Output Parser
- **Type / role:** `outputParserStructured`
- **Schema example:** `alertSeverity`, `alertMessage`, `immediateActions[]`, `stakeholdersToNotify[]`, etc.
- **Connections:** → `Budget Alert Agent Tool` (`ai_outputParser`)
- **Version notes:** typeVersion `1.3`.

#### Node: Routing Recommendation Agent Tool
- **Type / role:** `agentTool` — callable routing optimization specialist.
- **Input text expression:** same pattern with `$fromAI("costAnalysis", ...)`.
- **Output parsing:** yes via `Routing Output Parser`.
- **Model:** `Anthropic Model - Routing`.
- **Connections:**
  - Tool output → `Optimization Coordinator Agent` (`ai_tool`)
  - Receives `ai_languageModel` from `Anthropic Model - Routing`
  - Receives `ai_outputParser` from `Routing Output Parser`
- **Edge cases:** same `$fromAI(...)` dependency.
- **Version notes:** typeVersion `3`.

#### Node: Anthropic Model - Routing
- **Type / role:** `lmChatAnthropic`
- **Model:** `claude-3-7-sonnet-20250219`, temperature `0.2`
- **Connections:** → `Routing Recommendation Agent Tool` (`ai_languageModel`)
- **Version notes:** typeVersion `1.3`.

#### Node: Routing Output Parser
- **Type / role:** `outputParserStructured`
- **Schema example:** `routingStrategy`, `recommendations[]`, `totalPotentialSavings`, `implementationSteps[]`, etc.
- **Connections:** → `Routing Recommendation Agent Tool` (`ai_outputParser`)
- **Version notes:** typeVersion `1.3`.

#### Node: Cost Projection Calculator Tool
- **Type / role:** `toolCode` — JavaScript tool callable by coordinator to compute projections.
- **Input:** expects `query` to be a JSON string; code does `JSON.parse(query)`.
- **Expected fields (as coded):**
  - `current_cost`, `daily_average`, `days_elapsed`, `budget`
- **Outputs:** JSON string with burn rate, month-end projection, runway days, variance percent, status.
- **Connections:** Tool output → `Optimization Coordinator Agent` (`ai_tool`)
- **Edge cases / failures:**
  - If `query` is not valid JSON → parse throws and tool fails.
  - Division by zero if `days_elapsed` is 0; burn rate becomes Infinity/NaN.
  - Mismatch: upstream cost analysis uses fields like `totalCost`, `projectedMonthEndCost`; coordinator must map into the tool’s expected snake_case fields.
- **Version notes:** typeVersion `1.3`.

- “Optimization Coordinator — Aggregates all agent outputs and determines the optimal action plan…”

### Block 5 — Action-Type Routing → Slack/Email → Merge → Store Actions

**Overview:**  
Takes coordinator output and routes notifications: urgent Slack alerts or executive email reports. It merges notification execution results and stores the coordinator plan/actions to a Data Table.

**Nodes involved:**
- Route by Action Type
- Send Slack Alert
- Send Executive Report Email
- Merge Notification Results
- Store Optimization Actions

#### Node: Route by Action Type
- **Type / role:** `switch` — routes by coordinator `actionType`.
- **Rules:**
  - “Slack Alert” if `{{$json.actionType}} == "alert"`
  - “Executive Report” if `{{$json.actionType}} == "report"`
- **Fallback output:** `extra`
- **Connections:**
  - Input ← `Optimization Coordinator Agent`
  - Output → both `Send Slack Alert` and `Send Executive Report Email` (same output index in JSON; practically both are connected from the switch)
- **Edge cases / failures:**
  - If `actionType` differs (e.g., “email”, “slack”), it will go to fallback and may not notify anyone.
- **Version notes:** typeVersion `3.4`.

#### Node: Send Slack Alert
- **Type / role:** `slack` — posts alert message.
- **Authentication:** OAuth2 Slack.
- **Channel:** `channelId` is set via expression:  
  `{{ $('Workflow Configuration').first().json.slackChannel }}`
- **Message text:** formatted Slack markdown (includes priority, summaries, action list).
- **Connections:**
  - Input ← `Route by Action Type`
  - Output → `Merge Notification Results` (index 0)
- **Edge cases / failures:**
  - Invalid channel ID or missing OAuth scopes (e.g., `chat:write`).
  - Message formatting errors are rare, but very long messages may be truncated.
- **Version notes:** typeVersion `2.4`.

#### Node: Send Executive Report Email
- **Type / role:** `emailSend` — sends HTML executive report.
- **To:** `{{ $('Workflow Configuration').first().json.executiveEmail }}`
- **From:** placeholder sender address (must be replaced and supported by SMTP/Gmail settings).
- **Subject:** `AI Cost Intelligence Report - {{ $now.format('MMMM yyyy') }}`
- **HTML content:** large templated report referencing:
  - `$('Cost Intelligence Agent').first().json...`
  - `$('Optimization Coordinator Agent').first().json.recommendations` and `.actions`
- **Connections:**
  - Input ← `Route by Action Type`
  - Output → `Merge Notification Results` (indexes 0 and 1)
- **Edge cases / failures:**
  - **Schema mismatch risk:** Coordinator output parser example defines `coordinatedActions`, not `recommendations`/`actions`. If coordinator does not output those arrays, the email will show “No specific recommendations… / No actions required…”.
  - SMTP/Gmail auth failures, blocked sender, invalid from address, spam policies.
  - Expressions referencing `.json.potentialSavings` and `.json.insights` from `Cost Intelligence Agent`: these fields are not guaranteed by the output parser example; may render as 0/N/A text.
- **Version notes:** typeVersion `2.1`.

#### Node: Merge Notification Results
- **Type / role:** `merge` — combines results of Slack/email branches.
- **Mode:** `combine` with `combineByPosition`.
- **Connections:**
  - Inputs ← `Send Slack Alert` and `Send Executive Report Email`
  - Output → `Store Optimization Actions`
- **Edge cases / failures:**
  - If only one branch runs, combine-by-position can yield partial/odd pairing. In alert-only or report-only cases, consider using “Merge by key” or “Pass-through” patterns depending on desired persistence.
- **Version notes:** typeVersion `3.2`.

#### Node: Store Optimization Actions
- **Type / role:** `dataTable` — writes coordinator plan to audit log.
- **Target table:** `OptimizationActions`
- **Mapped columns:**
  - `priority`, `timestamp`, `actionType`, `budgetAlertSummary`, `coordinatedActions`, `notificationChannel`, `routingRecommendationSummary`
- **Connections:**
  - Input ← `Merge Notification Results`
- **Edge cases / failures:**
  - Data Table missing / schema mismatch.
  - `coordinatedActions` is an array; depending on Data Table column type, it may need stringification.
- **Version notes:** typeVersion `1.1`.

**Sticky note content (applies here):**
- “Alert & Reporting — Slack alerts for urgent issues; executive email report for leadership; results stored…”

## 4. Reproducing the Workflow from Scratch

1) **Create a new workflow**
